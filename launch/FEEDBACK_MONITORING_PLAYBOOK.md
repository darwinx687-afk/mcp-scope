# Feedback Monitoring Playbook

Do not turn every comment into a feature. Look for repeated friction, clear bugs, and evidence.

## First 24 Hours

- Check GitHub issues and comments.
- Watch for installation or command confusion.
- Track whether the README first screen answers basic questions.
- Record false positives, false negatives, and config shape requests.
- Note GitHub Action failures with run links.

## First 72 Hours

- Group feedback by class.
- Identify repeated wording confusion before changing product scope.
- Ask for redacted reports when needed.
- Keep security-sensitive details out of public threads.
- Summarize top three friction points in `FEEDBACK_TO_ROADMAP_REVIEW.md`.

## First 7 Days

- Review issue labels and duplicates.
- Decide whether docs fixes are enough.
- Prioritize low-risk parser/report improvements.
- Avoid adding live execution or cloud features without a consent design.

## Watch List

- GitHub stars.
- Issues and comments.
- Installation questions.
- Report confusion.
- False positives.
- False negatives.
- Config shape requests.
- GitHub Action failures.
- README confusion.
- Requests for live `tools/list` export.
- Requests for npm package.
- Requests for GitHub Marketplace action.

## Classification

- `bug`
- `false-positive`
- `false-negative`
- `docs-confusion`
- `config-shape-request`
- `integration-request`
- `feature-request`
- `security-sensitive`
- `not-in-scope`
