# 平台限制和表达方式

发布前先检查平台当前限制。字数、图片和格式规则会变化，不要靠旧印象。

## 首批中文渠道

### 掘金

按技术文章写。包含背景、命令、报告示例、局限和 GitHub 链接。使用 `launch/assets/release-banner-zh-CN.svg`。

### 小红书

写得像自然的项目记录，比掘金少一些技术细节。不要宣传感太重，不用个人照片，不用随机相册图片，也不用外部图片。使用 `launch/assets/social-card-square-zh-CN.svg`。

### 即刻

写成简短 builder update。直接、自然，以请大家反馈为主。使用 `launch/assets/social-card-zh-CN.svg`。

### 微信

微信群更短、更口语。朋友圈可以偏个人记录。只使用准备好的 zh-CN SVG 图片路径。

## 首批英文渠道

### GitHub Prerelease

只有最终人工审查后才手动发布。需要图片时使用 `launch/assets/release-banner-en.svg`。

### LinkedIn

用 builder update 的方式。重点讲问题、本地优先、preview 范围和希望收到反馈。使用 `launch/assets/social-card-en.svg`。

### X / Twitter

可以用一条短帖，也可以用短 thread。需要配图时使用 `launch/assets/social-card-en.svg`。

### Hacker News

触达高，但社区标准更严格。只有 GitHub prerelease 已公开、README 已人工检查后，再考虑 Show HN。默认纯文字，除非人工另行决定。

### Dev.to

按技术文章写，可以放在 LinkedIn/X 后，作为长尾内容。使用 `launch/assets/release-banner-en.svg`。

## 跳过或之后再说

### V2EX

V2EX 本轮跳过。保留 `launch/copy/v2ex.md` 作为归档，但不要发布。

### Reddit

Reddit 放到之后可选。只有明确人工批准并检查具体 subreddit 规则后再发。
